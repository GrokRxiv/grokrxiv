#RPT4 — Batching APIs + scheduling (exploration / future scope)

Status: exploration, not blocking RPT3. Reference only while RPT3's H2 validation finishes; lock decisions after RPT3 ships.

#Context

Per the operator's cost concern ($12+/day from a single build): API-only economics fail at arXiv volume (~500–2000 new papers/day on cs/math/physics combined). Three orthogonal levers help:

  1. Batch APIs — provider-side 50% discount on bulk submissions; 24h SLA.
  2. Off-peak scheduling — queue jobs at the operator's chosen time (e.g. nightly 02:00); spread work across hours.
  3. Durable queues — Postgres-backed worker pool so a paper-in-flight survives orchestrator restarts and can retry idempotently.

The CLI runner (operator's subscriptions) zeros marginal cost but is rate-limited by the subscription tier. The API path is unlimited concurrency but $/paper. Combining batch-API + CLI gives both: cheap fast on the API path, free on the CLI path, and the operator picks which based on urgency.

#Option matrix

#A. Provider Batch APIs (Anthropic + OpenAI + Gemini)

Provider Endpoint Max batch SLA Discount
Anthropic POST /v1/messages/batches 100K msgs / 256MB 24h 50%
OpenAI POST /v1/batches 50K msgs / 200MB 24h 50%
Gemini Vertex AI Batch Prediction (or Files API) 100K+ minutes–hours varies

Workflow:

  1. Submit a batch with N×6 messages (one per agent × N papers).
  2. Receive batch_id; poll status every 5 min.
  3. When ended, download the results JSONL, dispatch each to the verifier ladder + DB.

Per-paper cost on Anthropic batch (FP-RPT3 numbers × 0.5): API ingest+review goes from 0.30–0.30–0.30–0.50/paper to **0.15–0.15–0.15–0.25/paper**.

Cost: implementation ~6h. Mostly fits inside the existing LLMProvider trait by adding submit_batch(...), poll_batch(...), fetch_batch_results(...). The supervisor needs a batched code path that doesn't run the review DAG synchronously — instead enqueues a batch row, then a separate watcher (Option B/C) finalizes.

#B. In-process scheduler (tokio-cron-scheduler)

For "run every day at 02:00 local time": pure-Rust, ~300KB dep, runs inside the orchestrator process.

Pros: zero infra. Lives inside the existing grokrxiv daemon mode. Cons: state lost on restart; only one scheduler per process; no retries on missed firings.

CLI surface: grokrxiv daemon --schedule "0 0 2 * * *" (or --schedule "daily 02:00" with friendly parsing).

Cost: ~2h.

#C. Postgres-backed durable queue (river or apalis)

papers.jobs already exists per CLAUDE.md but is light. Promote it to a real durable queue.

river (PG NOTIFY/LISTEN, sqlx, opinionated): best fit if we accept the river job-row schema. Provides retries, dead-letter, scheduled-for, periodic jobs.

apalis (multi-backend incl. PG): more flexible, looser opinions. Same feature set.

Workflow:

  1. Scheduler (Option B) enqueues IngestPaperJob { arxiv_id, runner_kind, batch_mode } rows at 02:00.
  2. N worker processes (grokrxiv worker --concurrency 4) pull jobs, run the pipeline.
  3. Failed jobs retry with exponential backoff; permanent failures land in a dead-letter view.
  4. Operator can grokrxiv jobs list --status pending / --retry <id> / --cancel <id>.

Pros: jobs survive restarts; observable via Studio; trivial to scale by running more workers. Cons: real migration work; a second long-running process kind to operate.

Cost: ~10–14h.

#D. Temporal (durable workflow engine) — far future

If GrokRxiv eventually does multi-stage paper lifecycle (extract → review → revise → re-review → publish → respond-to-comments → repeat) with branches and retries, Temporal is the right primitive: deterministic replay, code-as-workflow, per-step observability, signal/query APIs.

Rust SDK exists (temporal-sdk-core + temporalio crate); needs the Temporal server (Docker or Cloud).

Cost: ~30h + ops burden of running Temporal. Defer to RPT6+ unless workflow complexity demands it.

#E. Combine with the CLI path

A subscription-tier claude CLI has a per-day quota. For 500 papers/day on the CLI path you must:

  • Stagger calls so we don't burst the quota — tokio_util::time::Throttle at e.g. 60 RPM.
  • Detect the CLI's rate-limit response and back off (the CLI returns specific error text on quota exhaust).
  • Fall back to API-batch (Option A) when the CLI quota is exhausted — gated by GROKRXIV_CLI_QUOTA_FALLBACK=batch.

This is the cheapest steady-state config: CLI handles most volume; batch-API absorbs overflow; nightly schedule runs both as background work.

#Acceptance for FP-RPT4 (a+b+c together)

Expected daily cost on 500 papers:

  • CLI portion (~80% if quota is generous): $0
  • API-batch portion (~20%, ~100 papers): 100 × 0.20=∗∗0.20 = **0.20=∗∗20/day**
  • Without batching/scheduling (today): 500 × 0.40=0.40 = 0.40=200/day

10× cost reduction at typical volume; 100% reduction once CLI quotas scale.

#Surface to ship

  • grokrxiv daemon --schedule <cron|friendly> — long-running scheduler process.
  • grokrxiv ingest <ids…> --batch — one-shot batch submission, prints batch_id.
  • grokrxiv batches status <id> — poll.
  • grokrxiv batches finalize <id> — dispatch results to verifier + DB.
  • grokrxiv worker --concurrency N — (FP-RPT5) PG-queue consumer.
  • grokrxiv jobs {list,retry,cancel} — (FP-RPT5) ops surface.

#Open questions to settle before locking FP-RPT4

  1. Where do scheduled runs source arXiv IDs from? The arXiv API supports from/to date ranges; a daily run could pull submittedDate >= yesterday. Or follow specific categories.
  2. What's the operator's actual claude/codex/gemini quota? Need to measure before tuning the throttle.
  3. Should batch results trigger the moderation queue immediately or wait for the moderator's morning? Probably the former — awaiting_moderation is the queue.
  4. PR creation per batched paper or one bundled PR? Per-paper preserves the current grokrxiv-reviews repo layout.

#Why this isn't part of RPT3

RPT3 is about making one review work correctly end-to-end (extraction → review → PR → web). Cost optimization is meaningful only after the per-review pipeline is reliable. Land RPT3 first, then RPT4 turns the validated pipeline into a sustainable production cadence.
