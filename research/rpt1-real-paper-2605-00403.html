#Context

The M1 smoke uses paper 2605.12484 and falls into "simulated PR" mode at the approve step because no GITHUB_TOKEN is set — that's why the smoke reports pr_url=https://github.com/GrokRxiv/reviews/pull/SIMULATED-... and assertion 9 fails.

#Phase 0.5 — Ingest enhancement: TeX source required (one-time enhancement, lands before the real-paper run)

Operator directive (2026-05-15):

  • TeX is the preferred review source; fetch + parse it for every paper that has it.
  • If a paper has no TeX source on arXiv (rare but real), fall back to PDF-based extraction (the current behavior).
  • PDF is always downloaded — it's the viewable artifact regardless of what the review reads from.
  • paper_assets tracks both tex_path (nullable — null means TeX not available) and pdf_path (always present).

Current state of crates/ingest/:

  • Fetches Atom metadata from export.arxiv.org/api/query for title/authors/abstract/category
  • Fetches PDF from the pdf_url in the Atom feed
  • Uses pdf-extract to convert PDF → text → regex-split into sections + bibliography
  • No source-bundle fetch. No TeX parsing.

Change set:

  1. Add source fetch — crates/ingest/src/download.rs (or new arxiv_source.rs): fetch https://arxiv.org/e-print/<id>. Returns either a single .tex file or a .tar.gz containing the LaTeX project. Use the same rate-limited semaphore that gates the PDF fetch (arXiv 3s/req policy).
  2. Add TeX parser — new crates/ingest/src/tex.rs:
    • If response is gzip+tar: extract to a temp dir, find the "main" tex file (heuristic: the .tex file containing \documentclass, OR the one named main.tex/paper.tex/<arxiv-id>.tex, OR the largest .tex file)
    • If response is a single .tex: use it directly
    • Parse via regex (pragmatic, not a full LaTeX parser): extract \title{...}, \author{...}, \begin{abstract}...\end{abstract}, \section{...} + \subsection{...} markers, \cite{...}/\bibitem{...} for citations
    • Resolve \input{...} and \include{...} recursively for multi-file projects (one level deep is enough for most papers)
    • Strip LaTeX comments (% at line start, anywhere after non-escaped %)
    • Strip LaTeX commands within section titles (\textbf{}, \emph{}, etc.) — keep the inner text
  3. Update pipeline::ingest() in crates/ingest/src/pipeline.rs:
    • Try to fetch source via the new module first
    • If source 200: build PaperExtract from TeX (sections, abstract, citations come from TeX)
    • If source 404 or unparseable: log a warning at INFO level (source=pdf_fallback paper=<id> reason=<...>), fall back to PDF-text extraction (current path)
    • PDF is always downloaded in parallel — needed for paper_assets.pdf_path (the human-viewable artifact)
    • Store both paths in paper_assets: tex_path (nullable) and pdf_path (existing column, NOT NULL)
    • Mark the extraction source on the PaperExtract struct: source_format: "tex" | "pdf" so downstream agents can know which one they got (and the verifier ladder can be less harsh on PDF-extracted bibliographies, which are typically lossier)
  4. Migration — supabase/migrations/20260515000003_paper_tex_path.sql:
  5. Tests — update crates/ingest/tests/integration.rs:
    • Mock the source endpoint with a small tar.gz fixture containing a known TeX file
    • Assert PaperExtract is built from TeX (title/abstract/sections come from the TeX, not the PDF)
    • Assert PDF is still downloaded but only used for paper_assets.pdf_path
  6. Backward compat — papers ingested under the old code path (e.g. 2c9d69c2-... from the FP6 M1 smoke) keep their PDF-derived PaperExtract. No retroactive re-ingest.

Document in docs/ingest-tex-with-pdf-fallback-applied.md:

  • What: ingest now fetches arXiv source bundles when available, parses TeX, builds PaperExtract from TeX content. When source is unavailable or unparseable, falls back to PDF-text extraction. PDF is always downloaded as a viewable artifact regardless of which path produced the extract.
  • Why: operator directive (2026-05-15) — TeX is the authoritative source for review content (cleaner sections, real citations, intact math); PDF is too lossy for math papers (verified empirically on 2605.00403, a math-ph paper with dense equations). PDF stays as the fallback so we don't lose coverage of source-less papers.
  • Risk: TeX projects with complex \input/\include chains beyond one level deep miss nested content. Some papers use unusual macros the regex misses; the verifier ladder + the source_format field surface these as low-confidence reviews rather than silent quality drops. Cost impact negligible (~50KB extra download per paper).
  • Reversal: revert the ingest crate + drop paper_assets.tex_path + drop the source_format field. No data loss — PDF path and behavior are unchanged.

After this enhancement is shipped (cargo build + cargo test green; the existing M1 paper 2605.12484 still re-ingestable into a fresh review_id with TeX-based extract that's comparable in quality), proceed to Phase 1.

Future-state note (out of scope this pass): arXiv also exposes an "HTML (experimental)" renderer at https://arxiv.org/html/<id> for many papers — pre-rendered semantic HTML with math as MathML, far cleaner than parsing raw TeX. Confirmed available for 2605.00403 (per the operator's screenshot). A later enhancement could prefer the HTML source when available and fall back to TeX, simplifying the parser. This pass stays on TeX-only per the operator directive; the HTML path is a follow-up.

#Phase 3 — Ingest + run the DAG

This will:

  • Hit export.arxiv.org/api/query?id_list=2605.00403 to fetch Atom metadata
  • Download the PDF (math-ph papers are PDF-only; no LaTeX source bundle)
  • Extract title/authors/abstract from XML, sections + bibliography from PDF via pdf-extract
  • Run the 6 specialists in parallel + meta_reviewer
  • Persist review row with status awaiting_moderation
  • Print review_id=<uuid>

Expected timing: ~90–120s wall (matches the M1 smoke's 92s). Expected cost: ~$0.19 (same envelope as 2605.12484).

Math-ph specific risks to watch for:

  • PDF text extraction may garble math symbols; sections like "3.1 Generalized Fourier Transform" may parse as headers but inline math gets lost. Acceptable degradation — the DAG still runs.
  • The citation agent may produce fewer citations than the ML paper did (math papers often have shorter bibliographies); still pass-verified.
  • The technical_correctness agent on Opus might run hotter on tokens because math claims need more reasoning. Watch tokens_out.

If ingest fails (network 5xx from arXiv, PDF unparseable), abort the run — no PR.

#Phase 5 — Approve to trigger the real publisher

With GITHUB_TOKEN set, this goes through the real-mode branch (crates/orchestrator/src/cli.rs:710-746), which:

  1. Reads artifacts from artifacts/<review_id>/ (review.html, review.md, review.tex, bundle.zip)
  2. Builds an octocrab client with the PAT token
  3. Calls publisher.open_review_pr(...) (crates/publisher/src/github.rs:62) — 6 sequential GitHub REST calls:
    • GET ref heads/main → base tip SHA
    • POST 4 blobs (one per file)
    • POST tree with the 4 blobs parented on main's tree
    • POST commit on that tree
    • POST ref refs/heads/review/2605.00403-<8charuuid>
    • POST pull (title=paper title, base=main, head=the new branch, body includes grokrxiv-review-id: <uuid> marker)
  4. Persists github_pr_url to the reviews row
  5. Prints pr_url=https://github.com/GrokRxiv/grokrxiv-reviews/pull/<N>
