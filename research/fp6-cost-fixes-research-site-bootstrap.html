#FP6 — Pipeline Cost Fixes + Research Site + Repo Bootstrap (Plus FP7/FP8 Scope Lock)

#Context

After FP5 shipped the architecture-of-record for a 2-tier hybrid, the operator asked three sharp questions:

  1. Were the cost estimates accurate? Yes — the measured $0.38/paper was real. BUT 97% was concentrated in two roles, and 56% was traced to a single implementation bug: meta_reviewer re-sends the full 52K-token paper extract alongside the 5 specialist outputs, when the prompt template only ever uses the specialist outputs. Pure waste.

  2. Why is the API route so expensive — isn't it enterprise-ready? The frontier APIs are enterprise-ready. Our implementation wasn't cost-optimized. Specifically: meta_reviewer bloat; same 52K paper extract stored 5× in review_agents.input_artifact JSONB (~260K tokens/paper of dup rows); Anthropic prompt cache enabled only on system messages, not user prompt; OpenAI/Gemini cache headers not wired at all; no review_cache table to skip re-runs.

  3. Can we wire a pricing API into an admin console? Yes — but not via CostGoat/AIPricing/PricePerToken (none expose a public API). Correct integration: LiteLLM's model_prices_and_context_window.json on GitHub — static JSON, no auth, ~500 models with input/output/cache pricing. Drop-in for FP8 pricing console.

Honest cost projection after FP6 cost fixes (API-only, no hybrid):

Metric Before FP6 After FP6 cost fixes After FP6 + 2-tier hybrid (FP5)
Per paper $0.38 ~$0.12 ~$0.0066 marginal
50/day MVP $6,935/yr ~$2,190/yr $120/yr marginal
Full arXiv (2,200/day) $305K/yr ~$96K/yr requires hybrid

The FP5 hybrid stays the right MVP architecture, but FP6 cost fixes are independent and benefit every tier — they land first.

Format shift introduced during planning:

The operator asked whether plan/research docs should be HTML instead of markdown ("Markdown is the draft. HTML is what humans read."). Decision: research artifacts get dual format — .md canonical + auto-generated .html artifact. Plan files stay .md while in flight, get an HTML mirror on ship. A standalone local Next.js viewer surfaces all HTML artifacts with search + index. A publish-plan skill + Stop-hook automates the markdown→HTML conversion when a plan ships.

Repo bootstrap also expanded into FP6 scope:

The repo is not yet a git repo. The operator asked to also: git init; add CLAUDE.md (Claude conventions for this repo); add SKILLS.md index pointing at .claude/skills/; add .claude/settings.json with hooks declarations; add README.md. These land in FP6 Track D as the foundation other tracks need.

Scope expansion from clarifications (already locked, but deferred to FP7/FP8):

  • 4 auth providers: Google, GitHub, Email magic link, ORCID → FP7
  • User/admin consoles → FP7
  • API console + pricing console (pricing + breakdown tabs) → FP8
  • Standalone HTML cost-breakdown doc for various Agent workloads (API/Subscription/SDK) → FP8
  • New agent abstraction — two modes: review_only (current) | review_and_revise (agents emit revision_artifact patches to LaTeX source → PR with per-artifact accept/reject checkboxes) → FP8

The plan splits into three FPs, each with a clear goal:

  • FP6 (build NOW, parallel agent team): cost fixes + HTML research artifacts + local Next.js viewer + repo bootstrap + publish-plan skill
  • FP7 (scope locked, build NEXT): auth + user/admin consoles
  • FP8 (scope locked, build AFTER): API console + pricing console + cost HTML doc + revision-mode agents

#Track B — HTML research artifacts (1 subagent)

Convert each research/*.md to a self-contained .html and keep both formats. Each HTML is:

  • Single file, opens via file:// in any browser
  • Inline CSS (dark theme, scannable, 8px baseline grid, ≥4.5 contrast)
  • Inline JS only where needed (collapsible sections, copy-to-clipboard on code blocks)
  • SVG diagrams (replace ASCII boxes with proper SVG)
  • Color-coded severity tags, syntax-highlighted code, rich tables
  • Self-contained head metadata (no external font/css/js refs)

Build pipeline:

  • Shared CSS template at research/_template/style.css (inlined at build time)
  • Node script research/_template/build.mjs reads every research/*.md, runs through remark + rehype + rehype-pretty-code + custom CSS injection, writes research/{name}.html
  • Re-runs automatically whenever a .md changes (Track E hook triggers it)
  • Idempotent — running twice produces byte-identical output

#Track E — Plan-ship hook + publish-plan skill (1 subagent, depends on Track D)

Automate the markdown→HTML flow for plans and any new research doc.

Skill: .claude/skills/publish-plan/SKILL.md

---
name: publish-plan
description: Publish a shipped plan or research doc to the research/ HTML artifacts and update the local research site index.
---

When invoked with a markdown file path:
1. Read the markdown file
2. Run `research/_template/build.mjs <path>` to generate the matching `.html` artifact in research/
3. Touch `research/site/lib/.search-index-dirty` so the next dev server reload regenerates the search index
4. Echo the resulting HTML path

Use cases:
- Agent has just edited `~/.claude/plans/fpN-*.md` to mark a pass Shipped
- Agent has just written/updated a `research/*.md` doc

Hook: .claude/settings.json → Stop hook

Script: .claude/hooks/maybe-publish-research.sh

  • Walks research/*.md and ~/.claude/plans/*.md
  • For each, compares mtime to the matching .html (if any)
  • If .md is newer (or no .html exists), runs the build script for that file
  • Outputs a short summary (Generated 2 HTML artifacts: X, Y)
  • Exits 0 on success; exits 1 on build failure (prevents silent rot)

Files:

  • .claude/skills/publish-plan/SKILL.md (new)
  • .claude/hooks/maybe-publish-research.sh (new, executable)
  • .claude/settings.json (extends from D4)
