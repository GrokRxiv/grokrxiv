#GrokRxiv — FIX PASS 4 (real typed DAG + responsive + residue cleanup)

Fresh plan; not an addendum to FP3. This pass closes the gaps codex flagged after FP3 shipped: the "review DAG" is actually a sequential six-call loop with loose-prompted JSON, the verifier ladder uses a permissive {"type":"object"} schema for every role, scheduler workers are stubs, render_to_disk writes a synthetic meta-review (not the real DAG outputs), multi-provider is adapter-complete but orchestration-incomplete, moderation queue is unpopulated, and there's residual disclaimer / nav / api-docs copy that contradicts the legal-only policy. Plus the frontend is not responsive on mobile.

#Context

The user-supplied codex audit + screenshot say plainly:

  • The architecture pitch (sources/context.txt:597) is a typed publication pipeline — parallel multi-agent review DAG with schema-constrained inputs/outputs, a meta-reviewer that synthesizes the five specialist outputs, a real verifier ladder per role, scheduler-driven ingestion, populated moderation queue, real artifacts, and a proven publish loop. The repo today only ships the first quarter of that.
  • Mobile rendering is broken (see iPhone-width screenshot of /about).
  • Stray disclaimer references remain in code/copy after FP2/FP3 supposedly removed them.

This pass converts the FP3 baseline into the architecture that the original sources/context.txt actually describes, with parallel agents owning chunks of the work.

#Track C — Real typed DAG (sequential dependency for D + E)

Owner: general-purpose agent, Rust focus.

  1. Per-role output schemas. Load the existing JSON Schemas under schemas/ at boot. The files exist from FP1:

    • schemas/summary_review.schema.json
    • schemas/technical_review.schema.json
    • schemas/novelty_review.schema.json
    • schemas/reproducibility_review.schema.json
    • schemas/citation_review.schema.json
    • schemas/meta_review.schema.json

    New AppState field agent_schemas: Arc<HashMap<AgentRole, serde_json::Value>> populated in from_config. Reuse the existing include_str!-style pattern; or read from $GROKRXIV_SCHEMAS_DIR ?? "schemas" once at boot.

  2. Per-role verifier ladders. Replace the single state.verifier: Arc<VerifierLadder> with state.verifiers: Arc<HashMap<AgentRole, VerifierLadder>>, each ladder built via VerifierLadder::standard(Some(schema_json)) against that role's output schema. Removes the "permissive object" workaround.

  3. JSON-schema-enforced LLM calls. The Claude provider already supports ResponseFormat::JsonSchema(...) (used by /preview). Switch the M1 supervisor's per-agent ChatRequest from ResponseFormat::Text to ResponseFormat::JsonSchema(schema_for_role). Tighten retry: on parse failure, one corrective retry with "Your previous output did not validate against the schema; return strict JSON." Drop the { "raw": resp.text } fallback — make a parse failure a hard error rounded to verifier_status='fail' with the raw output preserved in verifier_notes.parse_error.

  4. Parallel fan-out of the five specialists. Replace for (role, role_prompt) in roles.iter() with tokio::join! (or futures::future::join_all) over the five non-meta-reviewer roles. Cap concurrency with a tokio::sync::Semaphore::new(5) so we don't blow past Anthropic's per-key rate limit. Then run the meta-reviewer (step 5) once the join completes.

  5. Meta-reviewer = real synthesis node. Build a separate prompt that bundles the five specialists' JSON outputs as input artifact, asks the meta-reviewer to synthesize them into a MetaReview. Persist the bundle as the meta-reviewer's input_artifact (new column, see step 6). The current "6th independent call with the same paper prompt" path is removed.

  6. Persist input artifacts. New migration migrations/20250513000006_review_agent_inputs.sql: alter table review_agents add column input_artifact jsonb. Update db::insert_review_agent signature to take an input: serde_json::Value and persist it. The specialist agents' input is the PaperExtract-shaped object; the meta-reviewer's input is the 5-tuple of specialist outputs.

  7. Acceptance unit test. crates/orchestrator/tests/dag.rs (new) — with wiremock-stubbed Anthropic returning valid schema-compliant JSON for each role, runs run_one_paper_blocking against a fixture paper, asserts: 6 rows, all 6 have input_artifact set, all 6 have verifier_status='pass' against the role schema, the meta-reviewer row's input_artifact contains the five specialist outputs by role key.

#Track D — Real artifacts + scheduler + moderation lifecycle (depends on C)

Owner: general-purpose agent.

  1. render_to_disk reads real DB rows. Currently builds a synthetic MetaReview and vec![] as agents (supervisor.rs:282-310). Replace with: SELECT meta_review FROM reviews WHERE id=$1 → deserialize to MetaReview; SELECT role, model, output, verifier_status, verifier_notes FROM review_agents WHERE review_id=$1 ORDER BY role → map to Vec<AgentRecord>. Render against those. The render template already supports the full agent list via the agents context variable.

  2. Real scheduler workers. crates/orchestrator/src/supervisor.rs:

    • run_ingest: pull a paper from arXiv via the existing grokrxiv_ingest::pipeline::ingest; persist papers row; enqueue a Review job for it if submitted_date >= AUTO_REVIEW_FROM.
    • run_review: run the M1 DAG (now real per Track C); leave at awaiting_moderation.
    • run_publish: invoked only by admin approve; calls publisher::open_review_pr and persists the PR URL.
    • All three handle retries via the existing MAX_RETRIES + exp_backoff already wired in run_item.

  3. Moderation queue is populated. When insert_review lands a row at awaiting_moderation, also insert into moderation_queue (review_id, state) values ($1, 'pending'). CLI approve/reject/request_changes update the matching moderation row in addition to the review row. CLI withdraw/correct persist corrections rows + flip the review status.

  4. CLI commands work for real. Replace the stub eprintln!("...: DB update wiring — task #11.") lines in crates/orchestrator/src/cli.rs:486+:

    • reject <REVIEW_ID> --reason TEXT — updates moderation_queue.state, leaves review status at awaiting_moderation, prints rejected=<id>.
    • request-changes <REVIEW_ID> --notes TEXT — updates moderation_queue.state + notes.
    • withdraw <REVIEW_ID> --reason TEXT — inserts a corrections row of kind withdrawal, flips reviews.status = 'withdrawn', fires revalidate.
    • correct <REVIEW_ID> --rationale-md PATH — reads the markdown, inserts a corrections row of kind correction, flips reviews.status = 'corrected', fires revalidate.
    • list reviews [--status STATUS] and show <REVIEW_ID> — print real rows (already wired plumbing-wise; just stop emitting placeholders).

  5. m1-pipeline.sh extension: assert moderation_queue has a pending row after ingest, and after reject the row is rejected.
