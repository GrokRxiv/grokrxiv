The spec introduces ReviewAgent (role-level worker) + AgentRunner (execution backend). The operator wants ALL of this — phases 1–6 of the spec — built in one pass by a parallel agent team, with a real grokrxiv CLI replacing today's cargo run -- … and a public HTTP API mirroring it.

#Operator clarifications (locked)

  • One mega-pass, parallel-agent-built. Sub-agents build in parallel; integration is the only serial step.
  • All three local CLIs (claude, codex, gemini) are installed + authed on this machine.

#Provider + model: YAML/profile only — no ENV/CLI overrides

The operator's question: "should we have an env to set the agents types provider (claude, deepseek, codex, gemini, etc) and model type options, or is that set in our YAML and just keep there?"

Decision: keep provider: + model: in YAML (canonical) + TOML profile (per-deployment override). Do NOT add --provider / GROKRXIV_PROVIDER runtime overrides.

Reasons:

  • The TOML profile system already covers per-deployment switching (e.g. profile oss-tier-1 re-routes every role to local Ollama with different models).
  • Provider + model are structurally coupled — changing one usually requires changing the other. CLI flags inviting one-without-the-other are bug bait.
  • ENV-driven provider swaps are easy to leave on accidentally and cause surprising prod behavior.
  • Per-run experimentation is still possible via --model-for ROLE=MODEL (model swap only; provider stays from YAML).
  • Profile selection is itself overridable via --profile / GROKRXIV_PROFILE, so per-run "switch everything" is a single flag.

Provider enum expands (operator named DeepSeek explicitly): claude / openai / gemini / deepseek / xai / together / fireworks / groq / any other OpenAI-compatible. Implementation: extend crates/llm-adapter/src/providers/ with a generic OpenAICompatProvider taking (base_url, api_key, model) — the named providers (deepseek, groq, etc.) become thin factories that pre-fill the base URL. Anthropic and Gemini keep their dedicated providers due to incompatible request shapes.

#Consolidated runner taxonomy — exactly 4 types

Runner Concrete backends (selected by ENV) When

#Why Ollama over vLLM (operator's question)

Both support DeepSeek-V4, Qwen-2.5, Llama-3.x, etc. — at the model level they're interchangeable. The decision turns on environment fit:

Factor Ollama vLLM Winner
Mac M-series native ✓ first-class (brew app, Metal GPU) ⚠ vllm-metal is third-party, less stable Ollama
Daemon model management ✓ daemon swaps models on demand ✗ restart-per-load Ollama
Single-paper throughput adequate better (PagedAttention) tie at 50/day
Cloud GPU throughput adequate best-in-class vLLM (but ↓)
LiteLLM gateway support native native tie
Open WebUI compatibility first-class works via OpenAI shim Ollama
JSON-schema enforcement format: "json" (loose); strict via LiteLLM guided decoding (strict native) tie (we use LiteLLM either way)
Operational complexity one daemon + ollama pull Python + CUDA/MPS + restarts Ollama

Decision: standardize on Ollama as the only local_inference backend the runner directly knows about.

vLLM stays a deployment-time choice: in production, an operator can run vLLM behind LiteLLM and the orchestrator's LocalInferenceRunner doesn't notice — same OpenAI-compatible HTTP. This gives us Ollama's Mac ergonomics without giving up vLLM's cloud throughput.

LocalInferenceRunner itself is generic (an OpenAI-compatible HTTP client). The "Ollama-ness" lives only in the default ENV (OLLAMA_HOST=http://localhost:11434) and the doctor's reachability check. No --local-provider switch ships — just Ollama via direct URL or LiteLLM via GROKRXIV_LITELLM_URL.

Orthogonal SandboxPolicy (applied to cli or local_inference runners when isolation is desired):

Policy Behavior
none Run on host. Default.
container Wrap the runner in a Docker container (multi-arch arm64/amd64); RO-mount ~/.claude / ~/.codex / ~/.config/gemini; per-run workdir; --network=none unless role overrides.

Cloud runners are inherently sandboxed (the cloud provider runs the sandbox); they don't take a SandboxPolicy.

#Why this simplification works

  • claude_cli / codex_cli / gemini_cli are not really different runners — they're the same "spawn a subprocess, pipe prompt in, parse JSON out, validate against schema" with a different binary. One CliRunner impl + an ENV switch is enough.
  • e2b / vercel_open_agents are the same shape — "POST input to remote service, poll for output, parse JSON." One CloudRunner impl + an ENV switch.
  • ollama / vllm are HTTP clients to OpenAI-compatible endpoints. One LocalInferenceRunner + an ENV switch.
  • The original local_container "runner" was just CLI execution with isolation. That's a sandbox concern, not a runner concern.

#Side-effect boundary preserved

Supervisor still owns: DAG topology, concurrency, cache lookup, verifier ladder, DB persist, render lifecycle, publishing. Agents own only: role reasoning + structured output. Runners own only: execution mechanics.

#K.1 — Mac M-series local dev / local prod

Critical rule: never containerize the inference server on Mac. Docker on Mac has no GPU passthrough; running Ollama in Docker on Mac drops you to CPU-only inference, which is unusable for 32B-class models.

#K.2 — Cloud deployment

On cloud, inference IS containerized (Linux has GPU passthrough via nvidia-container-toolkit; CUDA images work). Topology:

The LocalInferenceRunner doesn't change — it's still an HTTP client. Only the URL it points at changes:

  • OLLAMA_HOST=http://ollama:11434 (in-VPC service)
  • Or via gateway: VLLM_BASE_URL=http://litellm:4000 (LiteLLM routes to the right backend)

For cloud OSS via managed services (no self-hosted GPU): set VLLM_BASE_URL=https://api.together.xyz/v1 or https://api.fireworks.ai/inference/v1 — both OpenAI-compatible — and the runner Just Works.

#K.3 — LiteLLM as the unified inference gateway

The operator's mention of LiteLLM is significant. Recommendation: wire LocalInferenceRunner to default to LiteLLM when present, and let LiteLLM handle the actual dispatch.

Why:

  • One stable internal URL (http://litellm:4000) regardless of which Ollama/vLLM/cloud-OSS-provider it routes to
  • Built-in retries, rate-limiting, fallback chains
  • Observability: per-request logs visible in the LiteLLM UI
  • Cost accounting: LiteLLM tracks tokens + estimated $ per call
  • Switching from Ollama to vLLM to Together.ai is a LiteLLM config change, not a runner code change

Wire-up:

  • ENV: GROKRXIV_LITELLM_URL=http://litellm:4000 (when set, takes precedence over OLLAMA_HOST / VLLM_BASE_URL)
  • LocalInferenceRunner checks GROKRXIV_LITELLM_URL first; falls back to direct provider URLs
  • infra/litellm/config.yaml ships with sensible defaults (model_list, fallbacks)

#CLI auth in containers

Operator's question: "should we just use current logged in cli auth tokens inside the rust container?" — yes, via RO mounts.

Host Container Mode
${HOME}/.claude/ /home/agent/.claude/ ro
${HOME}/.codex/ /home/agent/.codex/ ro
${HOME}/.config/gemini/ /home/agent/.config/gemini/ ro

Container runs as non-root agent (uid 1000). For pure-Rust testing without containers, the runner detects host CLIs and execs against host auth directly.

Risk + mitigation:

  • Compromised container payload could read mounted auth → --network=none default; per-role network opt-in
  • CLI tries to write session refresh to RO mount → refresh tokens last 30 days; host handles refresh between runs
